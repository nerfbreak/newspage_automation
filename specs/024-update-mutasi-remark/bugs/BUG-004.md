# Bug Report: Mutation Receiver Remark Overwritten by Previous Run

**Type**: Implementation drift
**Severity**: High
**Feature**: 024-update-mutasi-remark
**Reported**: 2026-07-12

## Description
When running Stock Mutation, the remark entered by the user for the Receiver (Penerima) is not captured correctly in the Newspage portal. Instead, the screenshot and final transaction show the remark from a previous bot execution (e.g., "Dari Depo Purwokerto" instead of the typed "TERIMA DARI PURWOKERTO"). This indicates an ASP.NET postback race condition where the Playwright script fills the remark text before the Reason Code dropdown's `AutoPostBack` response arrives, causing the server to overwrite the newly filled remark with its last known state.

## Artifact Traceability

### spec.md
- **Affected user story**: User Story 1 - Add Remark per Distributor
- **Affected requirements**: FR-002 (System MUST capture the value of "Remark Penerima"), FR-004 (System MUST pass the captured sender and receiver remarks to the playwright_engine)
- **Gap identified**: The automation implementation (playwright_engine.py) fails to persist the remark against ASP.NET postback lifecycles.

### plan.md
- **Affected sections**: `playwright_engine.py` update section
- **Impact**: Need to add explicit waits and DOM event triggers (`blur`/`change`) to ensure the remark field survives the UpdatePanel refresh.

### tasks.md
- **Affected tasks**: Update `run_mutasi_execution` to pass separate remarks
- **False completions**: None
- **Missing tasks**: Fix ASP.NET race condition in `_navigate_to_stock_adjustment`

## Root Cause Analysis
In `_navigate_to_stock_adjustment`, `dropdown.select_option(REASON_CODE)` triggers an ASP.NET postback. Immediately after, `remark_input.fill(remark_text[:50])` is called. The postback response then arrives, replacing the `UpdatePanel` contents with the server's state, erasing the newly filled remark text. Additionally, the character limit slicing was left at `[:50]` despite UI max_chars being updated to 100.

## Recommended Fix
1. Add `page.wait_for_timeout(1500)` after selecting the Reason Code to allow the postback to finish.
2. Update the truncation logic to `[:100]`.
3. Add `remark_input.press("Tab")` after filling to explicitly fire the `change` event.
