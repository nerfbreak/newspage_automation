# Bug Report: Mutation Table/Progress Alignment CSS Still Broken (BUG-009)

**Type**: CSS selector failure
**Severity**: Medium
**Feature**: 017-mobile-responsive
**Reported**: 2026-07-10
**Status**: Patched

## Description

BUG-003 and BUG-007 attempted to equalize DEDUCT/ADD table heights via CSS selectors targeting `.mutation-execution-layout-marker`. Both `+` (adjacent-sibling) and `~` (general-sibling) selectors fail to match in the live Streamlit DOM because wrapper depth and element ordering are inconsistent across renders.

Result: progress bars under the two tables are still vertically misaligned.

## Root Cause

Streamlit's DOM structure between the marker `<span>` and the `stHorizontalBlock` containing the tables is unpredictable. No CSS sibling selector can reliably bridge this gap across Streamlit versions.

## Recommended Fix

Stop relying on CSS selectors entirely. Inject `height: 400px` directly as an inline style on the `.neo-table-wrapper` div from Python. Add an optional `fixed_height` parameter to `render_responsive_dataframe()` in `utils.py`. The mutation page passes `fixed_height=400` when rendering the paired execution tables. This is DOM-structure-agnostic.

Changes:
1. `utils.py`: Add `fixed_height` param to `render_responsive_dataframe()`. When set, add inline `style="height: {fixed_height}px"` to the `.neo-table-wrapper` div.
2. `pages/4_stock_mutation.py`: Pass `fixed_height=400` in the two `render_responsive_dataframe()` calls for execution tables (lines 234, 239).
3. `static/style.css`: Remove the dead BUG-003/BUG-007 `@media` selector block (lines 1493-1503) since inline styles supersede it.
