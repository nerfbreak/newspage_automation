# Bug Report: Stock Mutation Table/Progress Bar Height Still Misaligned (BUG-003 Regression)

**Type**: CSS selector regression
**Severity**: Medium
**Feature**: 017-mobile-responsive
**Reported**: 2026-07-10
**Status**: Patched

## Description

BUG-003 added a CSS rule at line 1493-1500 in `static/style.css` to equalize DEDUCT and ADD table heights using a `.mutation-execution-layout-marker` adjacent sibling (`+`) selector. The fix is fragile because Streamlit wraps elements in varying numbers of `stElementContainer` divs depending on version and render context, causing the `+` combinator to miss.

Result: progress bars below the DEDUCT and ADD tables still render at different vertical positions when Description text wraps differently.

## Root Cause

The CSS relies on `div:has(.mutation-execution-layout-marker) + div[data-testid="stHorizontalBlock"]` (adjacent sibling). Streamlit 1.36+ inserts additional wrapper `stElementContainer` nodes between the marker and the `stHorizontalBlock`, breaking adjacency. The selector needs multiple fallback variants that may not cover all render scenarios.

## Recommended Fix

Replace the fragile `+` adjacent sibling selector with an ancestor `:has()` selector. Target the parent block container that **contains** the marker, then descend into `.neo-table-wrapper` children. This is DOM-structure-agnostic — it doesn't matter how many wrapper layers Streamlit inserts.

Specific approach:
1. Wrap the marker and execution columns in a single Streamlit `st.container()` with a `key` to create a scoped ancestor.
2. OR: Use a general sibling `~` selector instead of `+`.
3. OR: Use the Streamlit block-container parent `:has(.mutation-execution-layout-marker) .neo-table-wrapper` pattern.

Option 3 is simplest — no Python changes needed, CSS-only.
