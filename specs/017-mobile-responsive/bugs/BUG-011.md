# Bug Report: Mutation Execution Tables Have Asymmetrical Column Widths (BUG-011)

**Type**: Visual alignment / CSS
**Severity**: Low
**Feature**: 017-mobile-responsive
**Reported**: 2026-07-10
**Status**: Patched

## Description

While the overall table wrappers are now perfectly aligned at 400px (BUG-009), the *internal* layout of the DEDUCT and ADD tables is asymmetrical. The right table's columns (especially Description and Keterangan) stretch differently than the left table. This causes text to wrap differently and row heights to mismatch internally, creating a disjointed "ugly" visual experience despite the aligned borders.

## Root Cause

The HTML `.neo-table` class relies on the browser's default auto table layout algorithm. Because the content in the DEDUCT table (negative quantities) differs slightly from the ADD table (positive quantities), the browser calculates different column width proportions for both. The `width: 100%` rule exists, but without `table-layout: fixed`, columns grow/shrink dynamically based on their specific text length.

## Recommended Fix

Force identical column width proportions on `.neo-table` inside execution columns.
1. Apply `table-layout: fixed` to `.neo-table`.
2. Define explicit percentage widths for each column index using `nth-child` in `static/style.css` so both tables are forced to use the exact same grid scale, regardless of content variations (e.g., SKU 15%, Description 40%, Qty 10%, Status 15%, Keterangan 20%).
