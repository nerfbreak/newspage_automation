# BUG-004: Stock Mutation Upload Reset Runtime Error

## Summary

Clicking **Hapus File Upload** on the Stock Mutation page caused a Streamlit runtime exception because the inline handler directly assigned `None` to `st.session_state.mutasi_file_uploader` after the `st.file_uploader` widget had already instantiated and claimed ownership of that key.

## Root Cause

Streamlit 1.58 enforces widget-key ownership: once `st.file_uploader(key="mutasi_file_uploader")` runs, the framework owns that key for the remainder of the script body. The previous code used an `if st.button(...):` block that set `st.session_state.mutasi_file_uploader = None` inline — this ran *after* widget instantiation and violated the ownership contract.

## Fix

Replaced the inline branch with a named `_clear_mutasi_upload()` callback bound via `on_click=`. The callback:

1. Removes the uploader-owned key using `st.session_state.pop("mutasi_file_uploader", None)` — safe because callbacks run before widget re-instantiation on the next rerun.
2. Clears `mutasi_file_id` and `mutasi_review_df` so stale upload-derived state cannot persist.

No upload parsing, column mapping, automation logic, or Playwright selectors were changed.
