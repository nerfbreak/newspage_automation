# Bug Report: Active Streamlit Sessions Survive Password Rotation

**Type**: Untested flow
**Severity**: High
**Feature**: 019-session-invalidation
**Reported**: 2026-07-08
**Status**: Patched
**Patched**: 2026-07-08

## Description

Password rotation invalidates remembered-login cookies on a fresh app load, but an already authenticated Streamlit tab can keep `st.session_state.logged_in = True` and continue without re-checking the current `users_auth.session_version`. The user observed that the account effectively has to log out before the rotated password behavior is visible.

- **Affected requirements**: FR-001, FR-002, FR-003
- **Gap identified**: The spec covered remembered-cookie validation on app load, but did not explicitly require active Streamlit session state to be revalidated against credential-version metadata on rerun.

## Root Cause Analysis

Streamlit keeps per-tab session state independently from browser cookies. Once `logged_in` is true, the app did not revisit remembered-cookie validation, so password rotation could invalidate cookies without immediately invalidating an active server-side Streamlit session.

## Recommended Fix

1. Store the accepted `session_version` in `st.session_state.current_session_version`.
2. On each logged-in rerun, fetch the current user `session_version` from Supabase.
3. If the stored version is missing, stale, or unverifiable, delete `auth_user`, clear login state, and rerun to the sign-in screen.
4. Add smoke-test coverage proving `app.py` contains the active-session revalidation path.
